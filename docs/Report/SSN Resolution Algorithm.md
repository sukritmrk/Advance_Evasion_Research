## 1.Hell's Gate Algorithm
จาก Code snippet ของฟังก์ชัน `getHellsGateSSN` นี่คือลอจิกพื้นฐานในการดึง System Service Number (SSN) จากหน่วยความจำโดยตรง โดยอาศัยโครงสร้าง `EXPORT_TABLES` ที่เราสกัดมาได้จากขั้นตอนก่อนหน้า

ลอจิกการทำงานแบ่งเป็นสเต็ปดังนี้

![](../Images/Pasted%20image%2020260329225315.png)

ภาพ: Logic ของ Halo's Gate Algorithm

- **1. การค้นหาฟังก์ชัน** โค้ดทำการวนลูปตามจำนวน `dwNumberOfNames` เพื่อแปลง RVA เป็น String แล้วเทียบชื่อด้วย `ManualStrCmp`

- **2. การคำนวณ Absolute Address** เมื่อชื่อตรงกัน โค้ดจะดึงค่า Ordinal มาเป็น Index เพื่อหา RVA จาก `pdwAddressOfFunctions` จากนั้นบวกกับ Base Address ของ `ntdll.dll` เพื่อให้ได้ที่อยู่จริงของฟังก์ชัน (`uiFuncAddr`)

- **3. ตรวจสอบลายเซ็นคำสั่ง หรือ Pattern Matching** โค้ดทำการแปลงที่อยู่เป็น Pointer ระดับ Byte (`PBYTE`) แล้วเช็ค 4 Byte แรก ว่าตรงกับ Opcode ของ System Call Stub มาตรฐานของ Windows หรือไม่ ได้แก่ `4C 8B D1` (`mov r10, rcx`) และ `B8` (`mov eax, SSN`)

- **4. ดึงค่า SSN** หากตรง Pattern โค้ดจะกระโดดข้ามไป 4 Byte (`pByteAddr + 4`) แล้วทำ Pointer Dereference ดึงค่า SSN ขนาด 4 Byte (DWORD) ออกมาทันที

- **5. การรับมือ EDR Hook** หาก 4 Byte แรกไม่ตรง Pattern (เช่น ถูกแทนที่ด้วย `E9` หรือ `JMP`) โค้ดจะ Return `0` ทันที เพื่อบ่งบอกว่าฟังก์ชันนี้ไม่ปลอดภัยและถูก EDR ดักฟังอยู่

## 2.Halo's Gate Algorithm

![](../Images/Pasted%20image%2020260329225358.png)

ภาพ: การกำหนดขอบเขตการ Sorting SSN

   ในการทำงานของ Halo's Gate โค้ดจะต้องทำ **Neighboring Search** (การเดินส่องข้างบ้าน) ด้วยการบวกและลบ Pointer ( `+0x20` และ `-0x20` Bytes) ขยับขึ้นลงไปเรื่อยๆ เพื่อหา Syscall Stub ที่รอดพ้นจากการถูก EDR Hook แต่การ "เดินหน้าสแกน" ใน System Memory โดยไม่มีขอบเขตถือเป็นความเสี่ยงระดับร้ายแรง (Fatal Flaw) ในการพัฒนามัลแวร์ ด้วยเหตุผลดังต่อไปนี้:

   **1. หายนะจากการทำ Out-of-Bounds Read** ฟังก์ชัน API บางตัวอาจถูกจัดเก็บอยู่ตรงริมสุด (Edge) ของขอบเขตพื้นที่ Section `".text"` พอดี หากลอจิกการวนลูปของเราสั่งให้ Pointer เดินทะลุขอบเขตของ `".text"` ออกไป มันจะลุกล้ำเข้าไปยังพื้นที่หน่วยความจำส่วนอื่น เช่น:

   - **Unmapped Memory:** พื้นที่ว่างเปล่าที่ระบบปฏิบัติการยังไม่ได้จัดสรร

   - **Non-Executable Sections:** พื้นที่อย่าง `".data"` หรือ `".rdata"` ซึ่งมีสิทธิการเข้าถึง (Memory Protections) แตกต่างกัน

   **2. การเกิด EXCEPTION_ACCESS_VIOLATION (0xC0000005)** ทันทีที่ Pointer ของเราพยายามไปอ่านค่า (Dereference) จาก Memory Page ที่ไม่ได้รับอนุญาต โปรแกรมจะเกิด Access Violation และแครช (Crash) ทันที ในมุมมองของ Offensive Security การปล่อยให้ Payload แครชคือ **"ข้อผิดพลาดที่ยอมรับไม่ได้"** เพราะเมื่อโปรแกรมหยุดทำงานกะทันหัน ระบบปฏิบัติการ Windows จะสร้าง Crash Dump และส่ง Telemetry แจ้งเตือนไปยัง EDR ทันทีว่ามีโพรเซสต้องสงสัยพยายามทำ Memory Corruption ซึ่งเท่ากับเป็นการเผยตัวให้ Blue Team ทราบ

   **3. การออกแบบสถาปัตยกรรมเพื่อความปลอดภัยสูงสุด (Memory Security Design)** เพื่อปิดช่องโหว่นี้ โค้ดจึงถูกออกแบบให้ดึง `IMAGE_SECTION_HEADER` มาเพื่อ "กางแผนที่หน่วยความจำ" ก่อนที่จะเริ่มสแกน:

   - **`uiTextStart` (จุดเริ่มต้น):** คำนวณจาก `Base Address + VirtualAddress` ของ Section `".text"`

   - **`uiTextEnd` (จุดสิ้นสุด):** คำนวณจาก `uiTextStart + VirtualSize` (ขนาดพื้นที่ที่ใช้งานได้จริง)

   **ผลลัพธ์ด้านประสิทธิภาพ (Performance Impact):** ทุกครั้งก่อนที่ลอจิกจะทำการสแกนหา Pattern โค้ดจะประเมินเงื่อนไข `if (pUpFunc > uiTextStart && pUpFunc < uiTextEnd)` เสมอ การตรวจสอบนี้เป็นเพียงการเปรียบเทียบตัวเลขใน CPU Register ซึ่งมีความเร็วระดับ Clock Cycle (Overhead ต่ำมาก) แต่สามารถรับประกันเสถียรภาพ (Stability) ของมัลแวร์ได้ 100% ว่าการสลับ Context เข้าสู่ Kernel จะทำงานได้อย่างราบรื่นและไม่เกิดอาการ Memory Leak หรือ Crash กลางอากาศ
   
![](../Images/Pasted%20image%2020260329225421.png)

ภาพ: Logic หลักของ Halo's Gate Algorithm

การ "ส่องข้างบ้าน" ไม่ใช่การเดาสุ่ม แต่เป็นการใช้ประโยชน์จากโครงสร้าง System Memory ของ Windows ที่จัดเรียง Syscall Stub ไว้อย่างเป็นระเบียบ กลไกนี้ถูกออกแบบมาเพื่อ **ความเร็วสูงสุด** และ **ความปลอดภัยในหน่วยความจำ** โดยมีรายละเอียดทางเทคนิคดังนี้:

- **1. สถาปัตยกรรม Memory Alignment (ความลับของ `0x20`):** ใน `ntdll.dll` ฟังก์ชันระดับ Native API จะถูกจัดเรียง (Align) ให้มีระยะห่างกันพอดีที่ 32 Bytes (`0x20`) การใช้ Pointer Arithmetic บวก/ลบ ทีละ `0x20` จึงรับประกันได้ว่า Pointer จะตกไปที่จุดเริ่มต้น (Prologue) ของฟังก์ชันถัดไปหรือก่อนหน้าอย่างแม่นยำ โดยไม่ต้องเสียเวลา Parse Export Table ใหม่ ซึ่งช่วยลด Overhead ได้อย่างมหาศาล

- **2. ความเร็วในการประมวลผล (Raw Memory Access):** การเช็ค Pattern `4C 8B D1 B8` บน Pointer ที่ขยับไปมา เป็นการอ่านค่า System Memory โดยตรงผ่าน Raw Pointer (`*reinterpret_cast<DWORD*>`) ซึ่งประมวลผลด้วยความเร็วระดับ CPU Register โดยไม่มีการเรียกใช้ Windows API ใดๆ ที่อาจถูก EDR ตรวจจับ หรือดักฟังผ่าน User-land Hook

- **3. ตรรกะคณิตศาสตร์กู้คืน SSN (Sequential Logic):** สถาปัตยกรรม Windows x64 จะรันเลข SSN ตามลำดับการเรียงตัวใน Export Table
    
    - หากฟังก์ชันเป้าหมายถูก EDR ฝัง `JMP` (Hooked) โค้ดจะถอย Pointer ลง (`j = 1, 2, 3...`)
    
    - เมื่อเจอ `Syscall Stub` ที่สะอาด โค้ดจะดึง SSN นั้นมา แล้วใช้คณิตศาสตร์ย้อนกลับ
    
    - **ตัวอย่าง:** ถอยหลัง 2 ก้าว (`j=2`) เจอ SSN ลำดับที่ `0x15` ดังนั้น SSN ของเป้าหมายคือ `0x15 + 2 = 0x17`

- **4. Zero Memory Footprint:** กระบวนการทั้งหมดนี้ไม่มีการจองพื้นที่หน่วยความจำใหม่ (No `VirtualAlloc`) การคำนวณทั้งหมดเกิดและจบลงใน CPU Register ทำให้มีความปลอดภัยขั้นสุด ไม่ทิ้งร่องรอยให้ Heuristic Scanner ตรวจพบ และป้องกันอาการ Memory Leak ได้ 100%

## 3.Tartarus's Gate Algorithm

![](../Images/Pasted%20image%2020260329232712.png)

ภาพ: การกำหนดขอบเขตการ Sorting SSN

   ในส่วนของการกำหนดขอบเขต เป็นโค๊ดชุดเดียวกันกับที่ถูกใช้งานใน Halo's Gate ดังนั้นเราจะข้ามการอธิบายไปสู่ส่วนถัดไป

![](../Images/Pasted%20image%2020260329233106.png)

ภาพ: การกำหนดขอบเขตการ Sorting SSN

   แน่นอนว่า การใช้ Array ในการค้นหาชื่อของ SSN ของ Function ที่เราต้องการใช้ ก็ยังคงเป็นแบบเดิมกับที่ใช้ใน Halo's Gate  

![](../Images/Pasted%20image%2020260329233226.png)

ภาพ: Logic ส่วนการตรวจสอบหา Signature Pattern ของ Sorting Neighbor SSN ด้านบน (Up) ของ SSN เป้าหมาย 

   **1. ระบบตรวจจับ Hook เชิงรุก** ในยุคที่ EDR ไม่ได้จำกัดอยู่แค่การสแกน Signature กลยุทธ์หลักที่ Blue Team ใช้คือการทำ **Inline Hooking** ด้วยการแก้ไขชุดคำสั่งแรกของฟังก์ชัน (Prologue) โค้ดชุดนี้จึงออกแบบตัวแปร `upHooked` เพื่อทำหน้าที่เป็น "เครื่องสแกนทุ่นระเบิด" โดยอ่านค่า Raw Byte เพื่อหา Opcode ยอดฮิตที่เครื่องมืออย่าง Microsoft Detours หรือ EDR ชั้นนำมักใช้:

   - `0xE9` **(JMP) และ** `0xE8` **(CALL)** การกระโดดแบบ Near Jump 5-byte ซึ่งเป็นท่ามาตรฐานในการเบี่ยงเบนกระแสการทำงานไปยังโมดูลของ EDR

   - `0xFF 0x25` **(JMP QWORD PTR `[RIP+offset]`)** สถาปัตยกรรม x64 มักใช้ท่านี้ในการทำ Absolute Jump (14-byte Trampoline) เพื่อกระโดดข้ามขอบเขตหน่วยความจำที่ไกลกว่า 2GB

   - `0x68` **(PUSH) ตามด้วย** `0xC3` **(RET)** เทคนิคคลาสสิกในการยัด Address ของ EDR ลง Stack แล้วสั่ง Return เพื่อหลอกให้ CPU กระโดดไปหา

   - **ประสิทธิภาพ (Performance)** การตรวจสอบทั้งหมดนี้เกิดจากการทำ Pointer Dereference อ่านค่าหน่วยความจำทีละ Byte โดยตรง ซึ่งใช้รอบการประมวลผลต่ำมาก และหลีกเลี่ยงการใช้ API ใดๆ ทำให้มัลแวร์ทำงานด้วย **ความเร็วสูงสุด** และรักษาสถานะ Stealth ได้อย่างแนบเนียน


   **2. การตรวจสอบ `Syscall Stub` แบบเต็มโครงสร้าง (Full Signature Validation)** EDR บางค่ายพัฒนาเทคนิคไปถึงการทำ **Mid-function Hooking** หรือที่เป็นการฝัง Hook ไว้ตรงกลางฟังก์ชันแทนที่จะเป็นจุดเริ่มต้น การเช็คแค่ 4 Byte แรก (`4C 8B D1 B8`) จึงไม่การันตีว่า Loader ของเราจะเจอ Function ที่สะอาดอีกต่อไป

   - **Logic:** โค้ดชุดนี้จึงทำการตรวจสอบความสมบูรณ์แบบ 100% โดยการเจาะลึกลงไปที่ Offset `[18]`, `[19]` และ `[20]` (อ้างอิงตามโครงสร้าง `Syscall Stub` มาตรฐานของ Windows x64)

   - **Validation:** เป้าหมายคือการหา Opcode `0x0F 0x05` (คำสั่ง `syscall`) และ `0xC3` (คำสั่ง `ret`) ที่ส่วนท้ายของฟังก์ชัน

   - **ผลลัพธ์:** หากโครงสร้างตั้งแต่หัวจรดท้ายถูกต้องครบถ้วน จะเป็นการการันตีว่านี่คือ "พื้นที่ปลอดภัย" ที่เราสามารถดึงเลข SSN ไปใช้งานได้โดยไม่เสี่ยงต่อการเกิด Application Crash หรือการแจ้งเตือนจาก EDR

   **3.** เมื่อเรายืนยันได้แล้วว่า Neighbor Function นั้นสะอาดปราศจาก Hook กระบวนการชี้เป็นชี้ตายขั้นตอนสุดท้ายคือการคำนวณหา SSN ของเป้าหมายเดิมที่สูญเสียไป

   - **โครงสร้างของ Windows** ในกระบวนการ Compile ระบบปฏิบัติการ Windows จะเรียงลำดับฟังก์ชัน Native API และกำหนดเลข SSN เพิ่มขึ้นทีละ 1 ตามลำดับ

   - **The Logic:** โค้ดใช้วิธีดึงค่า SSN ของเพื่อนบ้านที่สะอาดออกมาผ่านตัวแปร `UpNeighborSSN` แล้วนำมาบวกด้วยค่า `j` ที่เป็นจำนวนระยะห่างของ Index ที่เราเดินย้อนกลับมาใน Export Address Table

   - ตัวอย่างแบบให้เห็นภาพชัด หากเป้าหมายของเราคือฟังก์ชัน A (ซึ่งโดน Hook) โค้ดเดินย้อนกลับไป 2 ลำดับ (`j=2`) และพบฟังก์ชัน C ที่สะอาด มีเลข SSN เป็น `0x15` โค้ดจะใช้สมการ `0x15 + 2` เพื่อสรุปว่า SSN ของฟังก์ชัน A คือ `0x17`

   - **ความเร็วและความปลอดภัย** การคำนวณทั้งหมดนี้ใช้ความซับซ้อนระดับ **O(1)** และเกิดขึ้นใน CPU Register ล้วนๆ ไม่มีการจองหน่วยความจำใหม่ หรือ Zero Allocation จึงป้องกันปัญหา Memory Leak และคงประสิทธิภาพระดับสูงสุดของระบบ System Memory ไว้ได้อย่างสมบูรณ์

![](../Images/Pasted%20image%2020260329233757.png)
   
   ภาพ: Logic ส่วนการตรวจสอบหา Signature Pattern ของ Sorting Neighbor SSN ด้านล่าง (Down) ของ SSN เป้าหมาย 
   
   **4. การค้นหาเพื่อนบ้านด้านล่าง** (Down neighbor SSN) ลอจิกส่วนนี้คือการทำงานคู่ขนาน หากการค้นหาด้านบน (Upward) ไม่สำเร็จ โค้ดจะสลับมาค้นหาฟังก์ชันในทิศทางลง (ลำดับ Index ที่สูงขึ้น) โดยยังคงรักษามาตรฐานความปลอดภัยสูงสุด:

   - **EAT Boundary Check** เงื่อนไข `if (i + j < ntdllExports.dwNumberOfNames)` คือหัวใจสำคัญด้าน **Memory Security** ทำหน้าที่บล็อกไม่ให้โปรแกรมวนลูปบวกค่า Index จนทะลุขอบเขตของ Export Address Table ซึ่งเป็นสาเหตุหลักที่ทำให้มัลแวร์เกิด Memory Corruption และ Crash

   - เนื่องจากการขยับ Index ลง (`+ j`) จะทำให้เราไปเจอเพื่อนบ้านที่มีค่า SSN _สูงกว่า_ ตัวเป้าหมาย ดังนั้นเมื่อโค้ดดึงค่า SSN ที่สะอาดมาได้ (`downNeighborSSN`) จึงต้อง **ทำการลบ (`- j`)** เพื่อทอนค่ากลับมาให้ตรงกับ SSN ของฟังก์ชันเป้าหมายเดิม

   - **Safe Exit & Error Handling** หากลอจิกทำงานจนสุดขอบเขตแล้วยังกู้คืนไม่ได้ โค้ดจะเลือก Return `-1` (เจอชื่อแต่กู้ SSN ไม่ได้) หรือ Return `0` (ไม่พบชื่อใน DLL) เพื่อให้โปรแกรมหลักทำงานต่อไปได้ (Graceful Exit) โดยไม่มีการฝืนอ่าน Memory อย่างผิดปกติ

   ภาพ: การค้นหา `"Zw"` หรือคำนำหน้าชื่อของฟังก์ชัน Native API และเริ่มการดึงรายชื่อ-จำนวน รวมถึง Address ของฟังก์ชันทั้งหมดที่เข้าข่ายมาเก็บไว้ใน Struct 

   จากภาพ โค้ดส่วนนี้คือขั้นเตรียมการของเทคนิค `FreshyCalls` ซึ่งเลิกพึ่งพาการอ่าน Signature/Opcode ที่เสี่ยงต่อการโดน EDR Hook แบบพลิกแพลง แต่หันมาใช้การดึง Address มาเก็บไว้แทน

   - **1. Stack-Based Array** การประกาศ `SyscallTable[500]` เป็น Array แบบ Fixed-size ทำให้หน่วยความจำถูกจองบน Stack ทันที การทำเช่นนี้ช่วยลด Overhead สู่ระดับศูนย์ เพราะไม่ต้องใช้ Heap Allocation ไม่ต้องเรียก `malloc` หรือ `new` ทำให้มีความเร็วสูงสุดและรับประกันได้ว่าจะไม่มี Memory Leak เกิดขึ้นในระบบ System Memory อย่างแน่นอน

   - **2. `"Zw"` Prefix** แทนที่จะเปรียบเทียบ String ยาว ๆ โค้ดใช้วิธีเช็ค Memory ระดับ Byte (`'Z'` และ `'w'`) ที่ต้นทางของ Pointer ชื่อฟังก์ชันทันที เนื่องจาก Native API มักจะขึ้นต้นด้วย `Nt` หรือ `Zw` (ใน Export Table มักใช้ `Zw`) การกรองแบบนี้ช่วยลดรอบการทำงานของ CPU ได้มากกว่า Algorithm แบบที่ผ่านมา

   - 3. เมื่อพบฟังก์ชันที่ขึ้นต้นด้วย `Zw` โค้ดจะดึง Ordinal -> RVA -> คำนวณเป็น Absolute Address (`uiFuncAddr`) จากนั้นนำ Pointer ของชื่อและ Address ไปเก็บสะสมไว้ใน `SyscallTable` พร้อมกับนับจำนวนด้วย `SyscallCount`
   
   หากสังเกตุดีดีจะพบว่าโค๊ดชุดนี้ใช้ตัวแปร Struct ที่ชื่อว่า `TempTable` ที่มีขนาดเทียบเท่า Array เพียง 1 ช่อง และ มีคุณสมบัติการจัดเก็บเหมือนกับ `SyscallTable` ซึ่งเราจะเอาไว้เป็นที่เก็บ Function Name และ Function Address สำหรับการสลับข้อมูล เดียวเราจะกล่าวในส่วนถัดไป

   ลอจิกชุดนี้มีความซับซ้อนเวลาเพียง O(N) ในการสแกน Export Table เป็นการแตะ System Memory เพียงแค่การอ่าน (Read-only) เท่าที่จำเป็น ถือเป็นการวางรากฐานข้อมูลที่รวดเร็วและปลอดภัย

   **กลไก `FreshyCalls` (The Sorting Technique) - ส่วนที่ 2 การเรียงลำดับและสกัด SSN**

   หลังจากที่เรารวบรวมรายชื่อและที่อยู่ (Address) ของฟังก์ชันที่ขึ้นต้นด้วย `Zw` มาเก็บไว้ใน `SyscallTable` ได้ทั้งหมดแล้ว โค้ดส่วนนี้จะทำหน้าที่ประมวลผลข้อมูลเหล่านั้นเพื่อสกัดเอาเลข SSN ออกมา โดยไม่ต้องไปยุ่งเกี่ยวกับ Opcode ภายในฟังก์ชันเลยแม้แต่ Byte เดียว:

   - **1. อัลกอริทึมการเรียงลำดับ หรือ Bubble Sort by Memory Address**
    
       - หัวใจสำคัญของ `FreshyCalls` อาศัยสัจธรรมของสถาปัตยกรรม Windows ที่ว่า **"Native API จะถูกจัดเรียงในหน่วยความจำตามลำดับของเลข SSN เสมอ"** ฟังก์ชันที่มีที่อยู่ (Address) ต่ำกว่า จะมีเลข SSN น้อยกว่าตามลำดับ
     
       - โค้ดใช้กลไก Bubble Sort ในการเรียงลำดับ Array `SyscallTable` โดยเปรียบเทียบ `FuncAddress` จากน้อยไปมาก
    
       - การเขียนเงื่อนไขลูป `SyscallCount - 1` และ `SyscallCount - i - 1` เป็นการลดรอบการทำงานของ CPU โดยตัดตัวที่เรียงเสร็จแล้วออกไป และยังช่วยอุดช่องโหว่ **Out-of-Bounds Memory Access** ได้ 100% ทำให้โปรแกรมไม่ Crash ขณะสลับตำแหน่งของตัวฟังก์ชันใน Array

   - **2. เทคนิคการข้าม Prefix
    
       - เมื่อได้ตารางที่เรียงลำดับตาม Address เรียบร้อยแล้ว โค้ดจะวนลูปอีกครั้งเพื่อค้นหาชื่อฟังก์ชันเป้าหมาย
    
       - **The Magic:** สังเกตการใช้พารามิเตอร์ `SyscallTable[i].FuncName + 2` และ `FuncName + 2` ในฟังก์ชัน `ManualStrCmp` นี่คือทริคที่ชาญฉลาดมาก! การบวก Pointer ไป 2 Byte คือการสั่งให้ข้ามตัวอักษร `Nt` หรือ `Zw` ทิ้งไป แล้วเปรียบเทียบแค่ชื่อแกนกลาง (เช่น ค้นหา `AllocateVirtualMemory` แทนที่จะหา `NtAllocateVirtualMemory`) ช่วยแก้ปัญหาความสับสนของ Prefix ใน Windows แต่ละเวอร์ชันได้อย่างหมดจด

-    **3. เมื่อเจอเป้าหมายแล้ว**
    
       - ทันทีที่ชื่อฟังก์ชันตรงกัน โค้ดจะทำการ `return i;` ทันที เพราะเมื่อ Array ถูกเรียงลำดับตาม Address แล้ว ลำดับ Index `i` ซึ่งเริ่มต้นจาก 0, 1, 2... จนถึง n จำนวน มันจะกลายเป็นเลข SSN ที่ถูกต้องของฟังก์ชันนั้นโดยอัตโนมัติ

**ความเหนือชั้นของ `FreshyCalls` ในแง่ Evasion** แม้การทำ Sorting จะใช้ทรัพยากรประมวลผลเพิ่มขึ้นเล็กน้อย แต่สิ่งที่ได้กลับมาคือ เทคนิคนี้ไม่ได้เข้าไปอ่าน Signature หรือ Opcode ภายในตัวฟังก์ชันเลย ต่อให้ EDR จะฝัง Hook (JMP) ลงไปลึกแค่ไหน หรือเปลี่ยนโครงสร้างฟังก์ชันไปอย่างไร `FreshyCalls` ก็จะไม่ได้รับผลกระทบใดๆ เพราะมันสนใจแค่ **"ที่อยู่เริ่มต้น (Base Address)"** ของฟังก์ชันเพื่อนำมาจัดเรียงเท่านั้น ถือเป็นเทคนิคที่เสถียรและหลบหลีก Heuristic Analysis ได้อย่างแนบเนียนที่สุด
