ในปัจจุบัน Cybersecurity มีความก้าวหน้าเป็นอย่างมาก จากเพียงรากฐานอย่างการสแกนไวรัสคอมพิวเตอร์แบบ Static (Signature-based) มาจนถึงการป้องกันขั้นสูงอย่าง Behavior Analytic ที่เป็นการป้องกันที่เจาะจงถึงพฤติกรรมของภัยคุกคามทางไซเบอร์ระดับสูง เพื่อตอบสนองต่อภัยคุกคามจากเทคนิคการเจาะระบบที่พัฒนาขึ้นทุกวันแบบก้าวกระโดด

ความเสียหายจากการโจมตีทางไซเบอร์ถือว่ามหาศาลมาก เพื่อให้เห็นภาพว่าเป็นปัจจุบันการโจมตีทางไซเบอร์และเป้าหมายของ Malware สมัยใหม่เป็นยังไง เราสามารถแบ่งเนื้อหาโดยสังเขปออกได้ 4 หัวข้อดังนี้

### 1. ความเสียหายทางการเงินโดยตรง (Direct Financial Loss)

นี่คือสิ่งที่เห็นชัดเจนที่สุดและเป็นเป้าหมายหลักของมัลแวร์สมัยใหม่ครับ

- **Ransomware** การจ่ายค่าไถ่เพื่อปลดล็อกข้อมูล หรือป้องกันไม่ให้ข้อมูลความลับถูกนำไปเผยแพร่ต่อสาธารณะชน 

- **Wire Fraud / Funds Theft** การหลอกลวงให้โอนเงิน หรือการเจาะเข้าระบบการเงินเพื่อขโมยเงินโดยตรง เช่น กรณีเจาะระบบ SWIFT ของธนาคารกลางของประเทศบังกลาเทศโดยฝีมือของกลุ่ม Hacker สัญชาติเกาหลีเหนือนามว่า Lazarus Group 

- **Intellectual Property Theft** การขโมยพิมพ์เขียว, Source Code, หรืองานวิจัยไปขายให้คู่แข่งหรือตลาดมืด

### 2. การหยุดชะงักของการดำเนินงาน (Operational Downtime)

มัลแวร์ที่ฝังตัวลึกและทำลายระบบจะสร้างผลกระทบข้อนี้หนักที่สุด

- **System Paralysis** ระบบเซิร์ฟเวอร์หรือเครือข่ายล่ม ทำให้ธุรกิจไม่สามารถให้บริการลูกค้าได้ เช่นกรณีของ Colonial Pipeline ที่ทำให้ท่อส่งน้ำมันในสหรัฐฯ ต้องหยุดชะงักลง

- **Lost Productivity** พนักงานไม่สามารถทำงานได้ตามปกติ อุปกรณ์เสียหาย ข้อมูลสำคัญสูญหาย ทำให้สูญเสียโอกาสในการสร้างรายได้ในแต่ละวันมหาศาล อีกทั้งยังส่งผลจากความน่าเชื่อถือของบริษัทในระยะยาว

### 3. ความเสียหายด้านชื่อเสียงและความเชื่อมั่น (Reputational Damage)

- **Loss of Customer Trust** ลูกค้าหมดความเชื่อมั่นและย้ายไปใช้บริการของคู่แข่ง โดยเฉพาะสถาบันการเงิน ธุรกิจการลงทุน โรงพยาบาล

- **Stock Price Drop** หุ้นของบริษัทในตลาดหลักทรัพย์มักจะร่วงลงทันทีที่มีข่าวว่าถูกแฮก และอาจใช้เวลาหลายเดือนกว่าจะฟื้นตัวกลับมาได้ ซึ่งเป็นผลกระทบมาจากการโจมตีใน 2 หัวข้อใหญ่ตามที่กล่าวไป

### 4. ค่าปรับทางกฎหมายและข้อบังคับ (Legal & Regulatory Fines)

- **Data Breach Penalties** เมื่อข้อมูลส่วนบุคคลของลูกค้าหลุดออกไป องค์กรจะต้องเผชิญกับค่าปรับมหาศาลจากกฎหมายคุ้มครองข้อมูลส่วนบุคคล เช่น GDPR ของยุโรป, PDPA ของไทย เป็นต้น

- **Lawsuits** การถูกฟ้องร้องจากลูกค้า ผู้ถือหุ้น หรือพาร์ทเนอร์ที่ได้รับผลกระทบจากการที่ข้อมูลหลุด เป็นผลที่ตามมาที่ร้ายแรงพอกับการถูกโจมตีทางไซเบอร์

### 1.ความเป็นมาของเทคนิคและปัญหา

ในโลกของ Offensive Security ปี 2026 การทำ Malware Development ไม่ได้ต่อสู้กับเพียงแค่ Signature-based Detection อีกต่อไป แต่เป็นการต่อสู้กับ EDR หรือ Endpoint Detection and Response ที่มีระบบ Behavioral Analysis และการตรวจสอบหน่วยความจำเชิงลึกที่ชาญฉลาดขึ้นอย่างน่ากลัว (แบบที่ดี) เลยเป็นปัญหาหลักที่ทำให้นักพัฒนาต้องเปลี่ยนผ่านเข้าสู่เทคนิคระดับสูง

ก่อนอื่นเรามาทำความเข้าใจถึงกลไกการตรวจจับและป้องกันที่สำคัญที่สุดในโลกเทคโนโลยีกันก่อน

### Endpoint Detection and Response หรือ EDR

มันคือเทคโนโลยีที่ออกแบบมาเพื่อฆ่าการโจมตีทางไซเบอร์ของแท้ กล่าวคือ ถ้า Antivirus (AV) รุ่นเก่าที่ทำการตรวจจับแบบ Signature-based ที่คอยเช็คแค่ว่าใครมีชื่ออยู่ในบัญชีดำ, ไล่สแกนไฟล์, ตรวจจับ String ใน .text section 

EDR ก็คือการนำการตรวจจับจาก Signature มาผสมเข้ากับ การตรวจพฤติกรรมที่ผิดปกติ หรือ Behavior Anomalies Analytic ที่คอยจับตาดูทุกความเคลื่อนไหวในระดับ Register ของระบบ เก็บพฤติกรรมภายในเครื่องที่ติดตั้ง EDR มาวิเคราะห์แล้วตัดสินว่า เป็น Malicious Software หรือไม่ หากใช่ก็จะสั่ง Kill/Terminate Process ทิ้ง

### อาวุธหลักของ EDR ที่เราจะกล่าวถึงและทำการ Bypass ตามวัตถุประสงค์ของบทความ มีดังนี้


### 1.1 User-land Hooking

นี่คือด่านแรกที่มัลแวร์ส่วนใหญ่จะจบลงตรงนี้ EDR จะทำการ Inject DLL ของตัวเองเข้าไปในทุก Process ที่รันอยู่ แล้วไปเขียนทับคำสั่งแรกๆ ของฟังก์ชันสำคัญใน `ntdll.dll` ด้วยคำสั่ง `JMP` หรือที่ภาษาของผู้ที่อยู่สายนี้เรียกสั้น ๆ ว่า Hooking 

- **หน้าที่** เมื่อ Malware เรียกฟังก์ชันอย่าง `WriteFile` แบบ Win API ปกติ หรือ เรียกฟังก์ชันผ่าน Native API โดนตรงจากการ `Direct Syscall` แทนที่การเรียกจะวิ่งไปที่ Kernel เพื่อส่ง System Service Number (SSN) เพื่อทำงานในฝั่ง Kernel เลย มันจะกระโดดไปที่ EDR ก่อน เพื่อตรวจสอบว่า 

- ใครเป็นคนเรียก?
- เรียกไปทำอะไร?
- พารามิเตอร์แปลกไหม? 

ถ้าตรวจสอบ Telemetry แล้วผ่าน EDR ก็จะให้ `Syscall` ข้ามไปฝั่ง Kernel (Ring 0) แล้วเรียกใช้งาน หากไม่ผ่าน Process ที่เรียกฟังก์ชันก็จะถูก Terminate ทันที นี้เป็นเพียง Behavior Analysis เบื้องต้นเท่านั้น ดูภาพประกอบด้านล่าง

### 1.2 Kernel Callbacks &`ETW-Ti`

นี่คือสิ่งที่ทำให้ Malware ในยุคปัจจุบันทำงานยากขึ้น โดย Windows มีกลไกที่ชื่อว่า **`ETW-Ti` (Event Tracing for Windows - Threat Intelligence)** ซึ่งถือว่ามันคือการจดบันทึกที่ Microsoft เปิดทางไว้ให้ผู้พัฒนา EDR โดยเฉพาะ

ทุกครั้งที่มีการใช้คำสั่ง `syscall` ไม่ว่าจะเป็นการเรียกแบบ Direct หรือ Indirect ก็ตาม ตัวของ Kernel จะบันทึกข้อมูลไว้เสมอ ข้อมูลนี้จะเป็นตัวบอกว่า `syscall` นี้ถูกเรียกมาจาก Memory Page ไหน

หาก EDR เห็นว่า `syscall` ถูกเรียกมาจาก Memory ที่ไม่มีที่มาที่ไป ไม่ได้มาจาก Disk หรือไฟล์ในเครื่อง แบบจู่ ๆ คำสั่งก็โผล่ขึ้นมากลาง RAM หรือที่ภาษาเทคนิคเรียกกันว่า Unbacked Memory และ Memory ที่ Protection สิทธิจัดการไฟล์เป็น `RWX` (Read-Write-Execute) มันจะส่งสัญญาณเตือนทันที

เกริ่นนำก่อนว่า แผนภาพนี้มีการป้องกัน โดยแบ่งพื้นที่การทำงานออกเป็น 2 ระดับชั้นอย่างชัดเจน คือ **Ring 3 Userland mode** (พื้นที่ทั่วไป) และ **Ring 0 Kernel mode** (แกนกลางระบบปฏิบัติการ)

### **Step 1 การโจมตีด้วย `Indirect Syscall`**
Malware ซึ่งอาศัยอยู่ในพื้นที่แบบ Unbacked Memory ทำการเรียกใช้เทคนิค **`Indirect Syscall`** (เพื่อยกตัวอย่างการ Bypass Userland Hooking จากข้อ 1.1) กระโดดข้าม Hook ของ EDR ในชั้น Ring 3

คำสั่งนี้โดดเข้าสู่ **Windows Kernel (`ntoskrnl.exe`)** ใน Ring 0 ได้สำเร็จ และพร้อมไปเจอกับดักของ EDR ที่วางไว้

### **Step 2 ด่านป้องกัน**
**จุดแยกระบบป้องกัน** ทันทีที่คำสั่งถึงมือ Kernel ระบบจะไม่ยอมประมวลผลและจะสั่งให้กลไกความปลอดภัย 2 ตัวทำงานคู่ขนานกัน

#### **กลไกที่ 1 Kernel Callback**

- 1. ระบบจะหยุดรอให้กลไกนี้ตรวจเสร็จก่อน ซึ่งนี้คือการทำงานแบบ **Synchronous**

- 2. `Parameter & Stack Analysis` เพื่อตรวจสอบความผิดปกติของ Parameter ที่ส่งเข้ามา

- 3. ถ้าน่าสงสัย ตัว Callback จะส่งเส้นสีแดง **`Deny`** กลับไปบล็อกคำสั่งจาก Malware ทันที

- 4. ถ้าแนบเนียนพอ ตัว Callback จะปล่อยผ่าน เกิดเป็นเส้นสีเขียว **`Granted`** วิ่งกลับขึ้นไปหา Malware แจ้งสถานะว่าทำงานสำเร็จ (Status 0x0) Malware จึงคิดว่าตัวเองปลอดภัยแล้ว

#### ขยายความการทำงานของ Kernel Callback

ภายใน `ntoskrnl.exe` จะมีตัวแปรที่เป็น Array ใช้เก็บ "รายชื่อ/ที่อยู่" ของฟังก์ชันที่ EDR ลงทะเบียนไว้ เมื่อมีคำสั่ง `Syscall` มุดลงมาถึง Kernel ระบบปฏิบัติการจะทำงานแบบ Synchronous ดังนี้

* 1. **Pause** Kernel จะหยุดการทำงานหลักไว้ก่อน

* 2. **Iterate** Kernel จะวนลูปอ่าน Array เพื่อเรียกฟังก์ชันของ EDR กรณีที่มีผลิตภัณฑ์รักษาความปลอดภัยมากกว่า 1 Kernel จะอ่าน Array ทุกตัวทุกค่ายที่ลงทะเบียนไว้

* 3. **Pass Information** Kernel จะส่ง Data Structure ที่ชื่อว่า `PS_CREATE_NOTIFY_INFO` ซึ่งเป็นข้อมูลของ `Callee` ไปให้ EDR อ่าน

* 4. **Analyze** EDR จะอ่านข้อมูลนี้ ถ้าพบว่าอันตราย มันจะแก้ตัวแปร `CreationStatus` ในโครงสร้างนั้นให้กลายเป็น `STATUS_ACCESS_DENIED`

* 5. **Execute** เมื่อ Kernel รับข้อมูลคืนมา ถ้าเห็น `CreationStatus` ที่โดนเปลี่ยนเป็น Denied มันจะสั่งยกเลิกการทำงานของคำสั่งนั้นทันที

EDR สมัยใหม่จะลงทะเบียน Callbacks อยู่ 4 ตัวหลัก เพื่อสกัดกั้นการโจมตี 
#### 1. `PsSetCreateProcessNotifyRoutineEx`

- **หน้าที่** ดักจับทุกครั้งที่มีการเรียก `NtCreateProcess` เช็ค Parent-Child Anomalies เช่น ถ้าอยู่ ๆ โปรแกรม `Word.exe` (Parent) สั่งเปิด `cmd.exe` หรือ `powershell.exe` (Child) มันจะบล็อกทันที เพราะผิดธรรมชาติของโปรแกรมนั้น ๆ
#### 2. `PsSetCreateThreadNotifyRoutine`

- **หน้าที่** ดักจับ `NtCreateThreadEx` ที่เป็นการสร้าง Thread ไม่ว่าจะในตัวเอง หรือยิงข้ามไป Process อื่น มันจะเล็งไปที่ Parameter ที่ชื่อ **`StartAddress`** เพื่อดูพิกัดของโค๊ดว่าจะ Run รันบนบรรทัดไหน ถ้าพิกัดนั้นชี้ไปที่ Unbacked Memory เมื่อไหร่ ตัว Process จะถูกระบุว่าเป็น Anomaly และถูก Terminate ทันที
#### 3. `PsSetLoadImageNotifyRoutine`

- **หน้าที่** ดักจับทุกครั้งที่ Process พยายามโหลดไฟล์ `.dll` หรือ `.exe` ลงใน Memory โดยจะเช็ค Certificate ของ DLL นั้นว่าเป็นของแท้ไหม ป้องกันการทำ DLL Sideloading หรือการฉีด DLL เถื่อน
#### 4. `ObRegisterCallbacks`

- **หน้าที่** ป้องกัน **LSASS** (Process เก็บรหัสผ่านของ Windows) มันจะทำการดักจับการขอสิทธิ Handle Token เช่น หากมี Malware ตัวนึงเรียกฟังก์ชัน `NtOpenProcess` เพื่อพยายามเข้าไปอ่านรหัสผ่านใน `lsass.exe` โดยขอสิทธิ์ `PROCESS_VM_READ` Callback ตัวนี้จะทำงาน EDR จะพิจารณาว่านี้ไม่ใช่โปรแกรมระบบ แล้วมันจะถอดสิทธิ์การอ่านของโปรแกรมนั้นทิ้ง ทำให้ Handle Token ที่ได้มากลายเป็นหมันทันที

#### **กลไกที่ 2 `ETW-Ti`**

- 1. ทำงานแบบ **Asynchronous** คือระบบจะไม่ขัดจังหวะการทำงานของ CPU ปล่อยให้ Malware ทำงานไประหว่างนั้นก็ไล่ตรวจสอบไป

- 2. Sensor ของ `ETW-Ti Provider` ที่ฝังอยู่ใน Kernel จะแอบเก็บประวัติการทำงานของ Stack หรือ **`Log Event + Call Stack`** เก็บไว้

  Sensor ของ `ETW-Ti` ไม่ได้จับทุกอย่างให้รกพื้นที่เซิร์ฟเวอร์ แต่มันเล็งไปที่ Anomalies Behavior โดยเฉพาะ โดยผูกติดกับฟังก์ชันใน Kernel

	- **การจัดการ Memory:** ฟังก์ชัน `NtAllocateVirtualMemory` จองพื้นที่ และ `NtProtectVirtualMemory` ใช้เปลี่ยนสิทธิ์แรมเป็น `RWX`
	
	- **Injection:** ฟังก์ชัน `NtWriteVirtualMemory` เขียนข้อมูลข้าม Process และ `NtCreateThreadEx`
	
	- **Credential Dumping:** เป็นการเรียก `NtReadVirtualMemory` โดยเป้าหมายคือ `lsass.exe`

  ทุกครั้งที่เรียกฟังก์ชันข้างบน `ETW-Ti` จะไม่แค่บอก EDR ว่ามีการเคลื่อนไหวที่ผิดปกติ แต่มันจะเรียกใช้ฟังก์ชันลับใน Kernel ชื่อ `RtlWalkFrameChain` เพื่อส่ง Call Stack ไปให้ EDR ด้วย เพื่อทำการตรวจสอบหา Unbacked Memory ต่อไป

### **Step 3 `ETW-Ti Session`**
ข้อมูล `Log Event + Call Stack` จะถูกส่งเข้าท่อ `ETW-Ti Session` แล้วส่งตรงขึ้นไปหาฝั่ง Userland โดยมีกฎเพียงข้อเดียวว่า เฉพาะโปรแกรม EDR ที่มีใบรับรอง ELAM Certified จาก Microsoft เท่านั้น ที่จะเข้ามาอ่านข้อมูลจากในท่อนี้ได้

  เหตุผลที่มัลแวร์ไม่สามารถเข้าไปยุ่งกับ `ETW-Ti` ได้ เป็นเพราะ Microsoft ล็อกระบบนี้ไว้ด้วยกุญแจ 2 ชั้น โดยจะเรียกว่า Sensor

   * **ELAM หรือ Early Launch Anti-Malware** คือโปรแกรมที่จะดึง Log จาก `ETW-Ti` ได้ ต้องมีลายเซ็นดิจิทัลระดับ ELAM จาก Microsoft เท่านั้น ซึ่งจะออกให้เฉพาะบริษัท EDR/Antivirus ที่ผ่านการตรวจสอบอย่างเข้มงวด

   * **PPL หรือ Protected Process Light** คือตัว Process ของ EDR ที่อ่านข้อมูลนี้ จะถูกป้องกัน PPL ทำให้ Malware ทั่วไปที่ต่อให้ Run ด้วยสิทธิ์ Administrator ก็ไม่สามารถเข้าไป Kill Process หรือแทรกแซงการอ่าน Log ของ EDR ได้

#### **Step 4 `Analyze`**
เมื่อ EDR บน Ring 3 ได้รับข้อมูล Call Stack มันจะทำการวิเคราะห์เชิงลึกอย่างละเอียด แม้คำสั่งจะวิ่งมาจาก `ntdll` อย่างถูกต้อง หากเมื่อสาวประวัติลึกลงไปจนถึงขั่วของ Stack ต้นทางที่แท้จริงกลับมาจาก **`Unbacked Memory`** ซึ่งเป็นพฤติกรรมของมัลแวร์ 100%

#### **Step 5 Terminate**
EDR จะส่งคำสั่ง **`Terminate Process`** เส้นสีแดงด้านบน กลับไปหา Malware ทันที

### 1.3 Stack Walking

EDR สมัยใหม่มีการทำ Heuristic scan หรือการสุ่มตรวจภายในหน่วยความจำ เป็นกับดักอีกชั้นที่เพิ่มขึ้นเพื่อตรวจสอบคำสั่งที่น่าสงสัยใน RAM เป็นระยะ

- **Stack Walking** เมื่อมีการรันฟังก์ชันที่น่าสงสัย EDR จะไล่เช็ค **Call Stack** (ลำดับการเรียกฟังก์ชัน) ว่าถอยหลังกลับไปเจอจุดเริ่มต้นที่สมเหตุสมผลไหม

* มันสามารถสแกนหา Code Pattern หาชุดคำสั่งที่ดูแปลกปลอมตามที่ถูกบันทึกในฐานข้อมูล เช่น รูปแบบการจัดเรียงคำสั่งการค้นหาไฟล์ DLL สำหรับเข้าถึงชื่อ Win API, การเปลี่ยนสิทธิ Protection ของ File-Process, การหารูปแบบของ Register ที่สลับไปมาหรือเรียงตัวผิดปกติเพื่อทำ `syscall`

- หาก Malware มีรูปแบบ Unbacked Memory เช่น ไม่ได้มาจาก `kernel32.dll` ตัว EDR ก็สามารถส่งสัญญาณไปหา SOC และ Terminate Process ของ Malware นั้นได้ทันที

- จากภาพตัวอย่างจะเห็นได้เลยว่า  เมื่อทุกครั้งที่โปรแกรมกำลังทำงาน มันจะนำข้อมูลการทำงาน 1 ครั้ง ไปวางไว้ที่จุดล่างสุดของ Stack แล้วเมื่อครั้งต่อไปทำงานจบก็จะนำมาวางทับซ้อนกันจาก **ล่างขึ้นบน** 
 
- การทำ Stack Walking คือการที่ EDR เดินสวนทางจาก **บนลงล่าง** ผ่าน Register ที่เรียกว่า RIP (Instruction Pointer) ซึ่งมีหน้าที่ชี้ไปที่ **Code Segment** หรือ "หน่วยความจำส่วนที่เก็บชุดคำสั่งโปรแกรม" แล้วไล่ตรวจสอบไปเรื่อย ๆ หากคำสั่งที่มีที่มาจาก Malware ที่ไม่มีการทำ `OpSec` หรือใช้เทคนิคที่ไม่รัดกุมมากพอ เช่น ไม่มีการ Obfuscation ลำดับการใช้งาน Register (ตรวจด้วย Code Pattern) ก็จะพบว่ามันมีลักษณะของ Unbacked Memory แบบที่เราคุยกันไปนั้นเอง

แน่นอนว่ายังไม่ทั้งหมด เรายังไม่ได้นับรวมมาตราการอย่าง **Intel CET (Control-flow Enforcement Technology), Intel PT (Processor Trace), SMEP & SMAP, VBS (Virtualization-Based Security)** ที่เป็นการป้องกันฝั่ง Hardware จาก CPU และ Kernel โดยตรง ซึ่งจะยังไม่ขอกล่าวถึงและอธิบายในบทความนี้

แต่ละด่านที่อธิบายไป ถือว่าเป็นการคัดกรองถึง 3 ชั้นใหญ่ เป็นกระบวนการสำคัญที่องค์กรขนาดกลาง-ใหญ่เกือบทุกองค์กร นำมาป้องกันระบบและเครือข่ายของตนเองผ่าน EDR หากปราศจากกระบวนการเหล่านี้ ลองจินตนาการดูว่า ความปลอดภัยของระบบจะกลายเป็นหายนะขนาดไหน

แน่นอนว่า การแก้ปัญหาของ Hacker หรือ Malware Developer ก็ต้องหาทาง Bypass การป้องกันเหล่านี้ให้ได้
